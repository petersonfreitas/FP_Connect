**Status:** Proposta inicial para atualização documental  

---

# 1. Objetivo

Este pacote consolida os documentos e backlogs pendentes do ecossistema FPWebTech após as decisões de arquitetura mais recentes.

A finalidade é padronizar os próximos módulos que ainda não estavam formalizados nos documentos do projeto e orientar futuras atualizações nos arquivos-base:

```text
DECISIONS.md
ROADMAP.md
ARCHITECTURE.md
MODULE_STATUS.md
backlog/
```

---

# 2. Decisões consideradas

| Decisão | Aplicação |
|---|---|
| Site institucional FPWebTech | Projeto isolado, fora do backlog operacional do ecossistema SaaS |
| Portal / Área do Cliente | Desconsiderado por enquanto |
| EixoGuard | Absorvido pelo futuro FP Router |
| FP Gateway | Backlog próprio e prioritário para integrações externas |
| FP Router | Baixa prioridade, complemento futuro do FP Tracking |
| FP Fiscal | Importante para evolução do FP Food |
| FP Sign | Backlog futuro para aceite, propostas e contratos simples |
| FP BI | Backlog/plano futuro para indicadores e dashboards cross-module |

---

# 3. Nomenclaturas oficiais

Usar os nomes curtos abaixo nos documentos oficiais:

```text
FP Gateway
FP Fiscal
FP Router
FP Sign
FP BI
```

Evitar títulos longos no nome do módulo, como:

```text
FP Gateway / Integrações
FP Sign / Aceite
FP BI / Dashboards
```

As descrições funcionais podem aparecer no conteúdo interno de cada backlog.

# 7. Observação de governança

Estes documentos não substituem os backlogs já aprovados. Eles funcionam como extensão do ecossistema e devem ser incorporados gradualmente, respeitando a ordem de execução já definida:

```text
Admin Console
FP Robots
FP Food
FP Tracking
FP Gateway
FP Fiscal
FP Sales / Marketing / Tickets / Billing
FP Sign
FP BI
FP Router
```

A ordem acima pode ser ajustada conforme necessidade operacional, mas recomenda-se manter o FP Gateway antes das integrações automáticas de pagamento, WhatsApp e Meta.
