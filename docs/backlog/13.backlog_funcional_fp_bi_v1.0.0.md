# 1. Visão Geral

O **FP BI** será o módulo responsável por consolidar indicadores, dashboards e relatórios do ecossistema FPWebTech.

Inicialmente, cada módulo pode ter seus próprios dashboards básicos. O FP BI deve surgir como uma camada central quando houver dados suficientes para análise cruzada entre módulos.

---

# 2. Objetivo do Produto

O objetivo do FP BI é oferecer visão executiva e operacional dos dados gerados pelos módulos da FPWebTech.

Exemplos:

```text
Funil comercial do FP Sales
Origem e conversão de leads do FP Marketing
Pedidos e faturamento operacional do FP Food
Entregas e performance do FP Tracking
Cobranças e inadimplência do FP Billing
Integrações e falhas do FP Gateway
Aceites e contratos do FP Sign
Indicadores fiscais do FP Fiscal
Rotas planejadas do FP Router
```

---

# 3. Princípio Central

```text
Os módulos são donos dos dados transacionais.
O FP BI é dono da leitura analítica e consolidação de indicadores.
O FP BI não altera regra operacional dos módulos.
```

# 5. O Que o FP BI Faz

O FP BI deve cuidar de:

```text
Consolidar indicadores entre módulos
Exibir dashboards executivos
Exibir dashboards operacionais
Gerar relatórios por empresa
Gerar visão global para Super Admin
Permitir filtros por período, empresa e módulo
Controlar permissões de acesso a indicadores
Exibir KPIs principais por módulo
Evoluir para camada analítica mais robusta no futuro
```

---

# 6. O Que o FP BI Não Faz

| Não deve fazer | Sistema responsável |
|---|---|
| Criar pedidos | FP Food |
| Alterar oportunidades | FP Sales |
| Rastrear entregas | FP Tracking |
| Emitir cobrança | FP Billing |
| Processar pagamento | FP Gateway |
| Emitir documento fiscal | FP Fiscal |
| Formalizar aceite | FP Sign |
| Planejar rota | FP Router |
| Criar automações | FP Robots |
| Corrigir dados transacionais | Módulo dono do dado |

# 13. Regras de Multiempresa

```text
Indicador sempre deve respeitar company_id.
Admin da Empresa só vê dados da própria empresa.
Super Admin pode ver visão global.
Usuário só vê indicadores dos módulos permitidos.
Dados de empresas diferentes não podem se misturar.
Módulo não contratado não deve aparecer para a empresa.
```

---

# 14. Critérios Gerais de Aceite

```text
FP BI possui catálogo inicial de indicadores.
Super Admin vê visão global.
Admin da Empresa vê apenas seus dados.
Indicadores respeitam módulos contratados.
Filtros por período funcionam.
Dados analíticos não alteram dados transacionais.
Dashboards básicos podem evoluir sem acoplar regras nos módulos.
```

---

# 15. Riscos e Cuidados

| Risco | Mitigação |
|---|---|
| Criar BI cedo demais | Começar com dashboards internos por módulo |
| Misturar dados entre empresas | Reforçar company_id e permissões |
| Acoplar relatório à regra operacional | Manter FP BI como leitura analítica |
| Escolher ferramenta prematuramente | Definir indicadores antes da ferramenta |
| Custos de infraestrutura | Começar simples, sem data warehouse dedicado |

---

# 16. Conclusão

O **FP BI** deve nascer como camada analítica do ecossistema, mas não precisa ser implementado antes dos módulos principais gerarem dados suficientes.

A melhor estratégia é iniciar com dashboards internos por módulo e evoluir para FP BI central quando houver necessidade real de visão cruzada entre vendas, marketing, food, tracking, billing, gateway, fiscal, sign e router.
