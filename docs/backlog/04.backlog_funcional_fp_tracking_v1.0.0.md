# 1. Visão Geral

O **FP Tracking** será o sistema especialista do ecossistema FPWebtech para execução operacional de entregas e rotas.

Ele será responsável por receber demandas de entrega, atribuir motoristas ou entregadores, registrar veículos, acompanhar status operacional, capturar localização em tempo real, gerar link público de rastreamento, registrar ocorrências e devolver atualizações aos sistemas de origem, como o **FP Food**.

O sistema será **genérico para entregas e rotas operacionais**, começando pelo caso de uso do FP Food, mas preparado para evoluir para outros cenários como coletas, entregas B2B, transporte leve, serviços externos e operações logísticas pequenas.

---

# 2. Objetivo do Produto

O objetivo do FP Tracking é permitir que empresas acompanhem a execução de entregas e rotas em tempo real, com visibilidade operacional para o gestor, uma tela simples para o entregador/motorista e um link público de rastreamento para o cliente final.

O sistema deve resolver três necessidades centrais:

```text
1. A empresa precisa saber quem está executando a entrega e em qual status ela está.
2. O entregador precisa de uma tela simples para iniciar, atualizar e finalizar a entrega.
3. O cliente final precisa acompanhar a entrega por um link público.
```

---

# 2.1 Diretriz de inicio aprovada

O FP Tracking deve entrar como modulo especialista de entrega e rastreio, usando o FP Food como primeiro consumidor real.

Diretriz operacional:

```text
Comecar pelo contrato Food -> Tracking.
Receber pedido pronto ou demanda de entrega do Food.
Devolver status e link de rastreio ao Food.
Manter a entrega simples do Food como fallback do MVP.
Publicar eventos tracking.* para FP Robots.
Deixar roteirizacao avancada para FP Router.
```

# 8. Conceitos Funcionais

## 8.1 Entrega

É a unidade operacional que precisa ser executada.

Exemplos:

```text
Pedido do FP Food
Entrega manual criada pelo operador
Coleta em cliente
Entrega B2B
Serviço externo
Transferência interna
```

### Regras

```text
Entrega deve pertencer a uma empresa.
Entrega deve ter origem e destino.
Entrega pode vir de sistema externo, como FP Food.
Entrega pode ser criada manualmente.
Entrega deve possuir status.
Entrega pode ter entregador atribuído.
Entrega pode ter link público de rastreamento.
```

---

## 8.2 Rota Operacional

É a execução agrupada de uma ou mais entregas.

No MVP:

```text
Uma entrega pode gerar uma rota operacional simples.
```

Na evolução:

```text
Uma rota pode conter várias entregas em sequência.
O Roteirizador poderá sugerir a melhor sequência.
```

---

## 8.3 Viagem / Execução

É o momento real em que a rota começa a ser executada.

Exemplo:

```text
Entregador João iniciou a entrega às 19:42.
```

### Regras

```text
Execução deve registrar início.
Execução deve registrar fim.
Execução pode registrar localização.
Execução pode registrar ocorrências.
Execução pode ser concluída com sucesso ou falha.
```

---

## 8.4 Motorista / Entregador

Pessoa responsável pela execução.

Pode ser:

```text
Motoboy
Motorista
Entregador próprio
Prestador externo
Operador de campo
```

### Regras

```text
Entregador deve pertencer a uma empresa.
Entregador pode estar ativo ou inativo.
Entregador inativo não deve receber nova entrega.
Entregador pode ter veículo vinculado.
Entregador pode ter entregas em andamento.
```

---

## 8.5 Veículo

Representa o meio de transporte usado na entrega.

### Dados funcionais básicos

```text
Identificação
Tipo de veículo
Placa, quando aplicável
Modelo, quando aplicável
Cor, quando aplicável
Status
Observações
```

### Tipos possíveis

```text
Moto
Carro
Bicicleta
Van
Caminhão leve
Outro
```

### Regras

```text
Veículo deve pertencer a uma empresa.
Veículo pode ser opcional em algumas entregas.
Veículo ativo pode ser usado em entregas.
Veículo inativo não deve ser atribuído a nova entrega.
```

---

## 8.6 Localização em Tempo Real

O FP Tracking deve nascer com localização em tempo real no MVP.

### Regras

```text
Entregador deve autorizar compartilhamento de localização.
Localização deve ser capturada pela tela mobile/PWA.
Localização deve ser vinculada à entrega/rota em andamento.
Cliente final deve visualizar apenas dados necessários.
Empresa deve acompanhar localização operacional.
Localização não deve ser exibida após o encerramento fora do contexto histórico permitido.
```

### Comportamento esperado

```text
Ao iniciar a entrega, o sistema começa a receber localização.
Durante a rota, a localização atualiza o painel operacional.
O link público exibe o acompanhamento da entrega.
Ao finalizar a entrega, a captura de localização deve parar.
```

---

## 8.7 Link Público de Rastreamento

URL pública para o cliente acompanhar a entrega.

Exemplo:

```text
tracking.seudominio.com.br/t/ABC123
```

### Regras

```text
Link deve ser único por entrega.
Link deve exibir apenas dados públicos.
Link deve funcionar sem login.
Link deve mostrar status atual.
Link deve mostrar localização quando disponível.
Link deve expirar ou ficar limitado após conclusão, conforme regra futura.
```

---

## 8.8 Ocorrência

Registro de problema ou evento operacional durante a entrega.

Exemplos:

```text
Cliente ausente
Endereço incorreto
Entrega recusada
Pedido avariado
Atraso
Problema com veículo
Problema com pagamento
Área de risco
Chuva/trânsito
Motorista sem contato
Outro
```

### Regras

```text
Ocorrência deve estar vinculada a uma entrega.
Ocorrência pode ser registrada por operador ou entregador.
Ocorrência deve ter tipo e descrição.
Ocorrência pode impactar status da entrega.
Ocorrência deve aparecer no histórico.
```

# 11. Regras de Negócio

## 11.1 Regras de Entrega

```text
Toda entrega deve pertencer a uma empresa.
Toda entrega deve possuir origem.
Toda entrega deve possuir destino.
Toda entrega deve possuir status.
Entrega pode vir do FP Food.
Entrega pode ser criada manualmente.
Entrega cancelada não pode ser iniciada.
Entrega entregue não deve ser alterada sem permissão especial.
```

---

## 11.2 Regras de Atribuição

```text
Entrega pendente pode receber entregador.
Entregador precisa estar ativo.
Entregador não deve receber entrega se estiver indisponível, conforme regra da empresa.
Veículo pode ser vinculado à entrega.
Veículo inativo não deve ser atribuído.
Atribuição deve aparecer no histórico.
```

---

## 11.3 Regras do Entregador

```text
Entregador visualiza apenas entregas atribuídas.
Entregador deve iniciar entrega antes de enviar localização operacional.
Entregador pode registrar ocorrência.
Entregador pode finalizar entrega.
Entregador não altera valor, pagamento ou dados comerciais do pedido.
```

---

## 11.4 Regras de Localização

```text
Localização em tempo real entra no MVP.
Localização deve depender de autorização do entregador.
Localização deve estar vinculada a uma entrega/rota em andamento.
Localização deve ser exibida no painel operacional.
Localização deve ser exibida no link público de forma adequada.
Localização deve parar após finalização, cancelamento ou falha definitiva.
Falha de localização não deve impedir atualização manual de status.
```

---

## 11.5 Regras de Link Público

```text
Toda entrega pode ter link público.
Link deve ser único.
Link deve exibir apenas dados públicos.
Link não deve permitir alteração da entrega.
Link deve exibir status atualizado.
Link pode exibir mapa/localização.
Link deve exibir conclusão, cancelamento ou falha.
```

---

## 11.6 Regras de Ocorrência

```text
Ocorrência deve ter tipo.
Ocorrência pode ter descrição.
Ocorrência deve ficar no histórico.
Ocorrência pode ser criada por operador ou entregador.
Ocorrência pode emitir evento.
Ocorrência crítica pode destacar a entrega no painel.
```

---

## 11.7 Regras de Integração com FP Food
