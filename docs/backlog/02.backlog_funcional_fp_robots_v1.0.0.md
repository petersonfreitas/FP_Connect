# 1. Visão Geral

O **FP Robots** será o módulo operacional de integrações e automações do ecossistema FPWebtech.

Ele será responsável por receber eventos dos sistemas especialistas, validar a origem, aplicar regras de automação, executar ações, enviar e-mails, disparar webhooks, integrar sistemas internos, registrar logs, permitir retry e reprocessamento de falhas.

O FP Robots não será o painel administrativo geral do ecossistema. Essa responsabilidade pertence ao **FP Connect Admin Console**.

---

# 2. Objetivo do Produto

O objetivo do FP Robots é permitir que os sistemas do ecossistema FPWebtech se comuniquem entre si e com sistemas externos de forma padronizada, segura, auditável e reprocessável.

Exemplo:

```text
CRM aprova uma proposta
↓
CRM emite evento crm.proposal.approved
↓
FP Robots recebe o evento
↓
Identifica regras configuradas
↓
Envia e-mail ao financeiro
↓
Chama webhook/API interna do Reembolso
↓
Registra logs e execuções
↓
Permite reprocessar falhas
```

---

# 3. Princípio Central

```text
O sistema de origem decide o que aconteceu.
O FP Robots decide quais automações executar.
O sistema de destino executa sua própria regra de negócio.
```

# 8. Conceitos Funcionais

## 8.1 Evento

Evento é um fato ocorrido em um sistema de origem.

Exemplos:

```text
crm.proposal.approved
routes.delivery.completed
menu.order.created
eixoguard.finding.created
billing.subscription.overdue
```

Um evento não é uma regra, nem uma ação. Ele apenas informa que algo aconteceu.

---

## 8.2 Catálogo de Eventos

Lista os eventos permitidos no ecossistema.

Cada evento deve possuir:

```text
Nome funcional
Código do evento
Sistema de origem
Descrição
Status
Versão
Indicação se está ativo ou inativo
```

---

## 8.3 Regra de Automação

Define o que deve acontecer quando determinado evento for recebido.

Exemplo:

```text
Quando receber crm.proposal.approved:
- Enviar e-mail ao financeiro
- Chamar API interna do Reembolso
- Registrar execução
```

Uma regra pode gerar zero, uma ou múltiplas ações.

---

## 8.4 Ação

Ação é a execução gerada por uma regra.

Tipos de ação do MVP:

```text
Enviar e-mail
Enviar webhook
Chamar API interna
Registrar notificação interna
Registrar log técnico
```

Tipos pós-MVP:

```text
Enviar WhatsApp
Criar evento Outlook
Criar tarefa interna
Enviar push notification
Executar workflow encadeado
```

---

## 8.5 Conector

Conector é a configuração funcional usada para executar uma ação.

Conectores do MVP:

```text
SMTP
Webhook
API interna entre sistemas
```

Conectores pós-MVP:

```text
WhatsApp
Outlook / Microsoft Graph
ERP externo
Google Calendar
SMS
Push notification
```

---

## 8.6 Execução

Execução representa uma tentativa real de executar uma ação.

Exemplo:

```text
Ação: enviar webhook para Reembolso
Status: falhou
Erro: timeout
Tentativa: 2 de 5
Próxima tentativa: em 5 minutos
```

---

## 8.7 Reprocessamento

Reprocessamento é a tentativa manual ou automática de executar novamente uma ação que falhou.

No MVP:

```text
Super Admin pode reprocessar qualquer falha.
Admin FPWebtech pode reprocessar conforme permissão.
Admin da Empresa pode reprocessar falhas da própria empresa.
```

# 9. Fluxos Funcionais

## 9.1 Fluxo — Receber Evento

```text
Sistema origem emite evento
↓
FP Robots recebe o evento
↓
Valida origem
↓
Valida empresa
↓
Valida se o evento está permitido
↓
Identifica duplicidade
↓
Registra evento
↓
Identifica regras aplicáveis
↓
Gera ações
```

### Critérios de aceite

```text
Evento sem origem válida deve ser recusado.
Evento de empresa inativa deve ser recusado ou bloqueado conforme regra.
Evento não permitido deve ser recusado.
Evento duplicado não deve gerar execução duplicada.
Evento válido deve ser registrado.
Evento válido deve disparar regras ativas compatíveis.
```

---

## 9.2 Fluxo — Executar Ação de E-mail

```text
Evento recebido
↓
Regra identifica ação de e-mail
↓
Sistema seleciona conector SMTP
↓
Sistema seleciona template
↓
Sistema aplica variáveis do evento
↓
Sistema envia e-mail
↓
Sistema registra sucesso ou falha
```

### Critérios de aceite

```text
E-mail só deve ser enviado se houver conector SMTP válido.
Template deve aceitar variáveis do evento.
Falha de envio deve gerar log.
Envio com falha pode ser reprocessado.
Envio com sucesso deve registrar evidência.
```

---

## 9.3 Fluxo — Executar Webhook

```text
Evento recebido
↓
Regra identifica ação webhook
↓
Sistema monta payload
↓
Sistema envia requisição para URL configurada
↓
Sistema registra resposta HTTP
↓
Sistema marca ação como sucesso ou falha
```

### Critérios de aceite

```text
Webhook deve possuir URL de destino.
Webhook pode possuir headers configurados.
Resposta do destino deve ser registrada.
Erro HTTP deve gerar falha.
Webhook com falha pode ser reprocessado.
```

---

## 9.4 Fluxo — Executar Ação Interna entre Sistemas

```text
Sistema origem emite evento
↓
FP Robots recebe evento
↓
Regra identifica ação interna
↓
FP Robots chama sistema destino
↓
Sistema destino executa sua própria regra
↓
FP Robots registra resposta
```

### Exemplos

```text
CRM → Reembolso
Cardápio/Delivery → Rotas/Tracking
Rotas/Tracking → EixoGuard
Roteirizador → Rotas/Tracking
Billing → FP Connect Admin Console
```

### Critérios de aceite

```text
FP Robots deve apenas encaminhar a solicitação.
Sistema destino deve executar a própria regra.
Falha do sistema destino deve gerar log.
Ação interna deve ser reprocessável.
Resposta do sistema destino deve ser registrada.
```

---

## 9.5 Fluxo — Reprocessar Falha

```text
Ação falha
↓
Sistema registra erro
↓
Sistema aplica retry automático, se configurado
↓
Falha permanece pendente ou definitiva
↓
Usuário autorizado acessa logs
↓
Usuário solicita reprocessamento
↓
Sistema executa ação novamente
↓
Nova tentativa é registrada
```

### Critérios de aceite

```text
Reprocessamento deve manter histórico de tentativas.
Admin da Empresa pode reprocessar falhas da própria empresa.
Admin da Empresa não pode reprocessar falhas de outra empresa.
Usuário deve visualizar erro antes de reprocessar.
Nova execução deve registrar novo status.
```

---

## 9.6 Fluxo — Visualizar Logs com Payload Mascarado

```text
Admin da Empresa acessa logs
↓
Sistema lista apenas eventos da própria empresa
↓
Admin abre detalhe do evento
↓
Sistema exibe payload com dados sensíveis mascarados
↓
Admin visualiza status e falhas
↓
Admin pode reprocessar, se autorizado
```

### Critérios de aceite

```text
Admin da Empresa só visualiza logs da própria empresa.
Payload sensível deve ser mascarado.
Credenciais nunca devem ser exibidas.
Logs devem mostrar status, erro e tentativas.
Reprocessamento deve respeitar escopo da empresa.
```

---

## 9.7 Fluxo — Testar Conector

```text
Usuário autorizado acessa conector
↓
Clica em testar
↓
Sistema executa teste controlado
↓
Sistema exibe sucesso ou erro
↓
Resultado do teste é registrado
```

### Critérios de aceite

```text
SMTP pode ser testado.
Webhook pode ser testado.
API interna pode ser testada.
Falha deve exibir mensagem objetiva.
Teste não deve executar regra real de negócio indevida.
```

# 14. Regras de Negócio

## 14.1 Regras de Evento

```text
Todo evento deve ter origem identificável.
Todo evento deve pertencer a uma empresa.
Evento deve estar no catálogo de eventos.
Evento de origem não autorizada deve ser recusado.
Evento duplicado não deve gerar execução duplicada.
Evento válido deve ser registrado.
Evento válido pode gerar zero, uma ou múltiplas ações.
Evento inválido deve retornar erro claro ao sistema origem.
```

---

## 14.2 Regras de Automação

```text
Regra ativa pode gerar ações.
Regra inativa não gera ações.
Regra pode ser global ou específica por empresa.
Regra global só pode ser criada por perfil autorizado.
Regra pode ter condições simples.
Regra pode ter uma ou múltiplas ações.
Alteração de regra deve ser auditada.
Regra não deve executar lógica profunda do sistema origem ou destino.
```

---

## 14.3 Regras de Ação

```text
Toda ação deve possuir status.
Ação pendente deve ser processada.
Ação com sucesso deve registrar evidência.
Ação com falha deve registrar erro.
Ação pode ter retry automático.
Ação pode ir para falha permanente após limite de tentativas.
Ação pode ser reprocessada manualmente por usuário autorizado.
Ação ignorada manualmente exige justificativa.
```

---

## 14.4 Regras de Reprocessamento

```text
Super Admin pode reprocessar qualquer falha.
Admin FPWebtech pode reprocessar conforme permissão.
Admin da Empresa pode reprocessar falhas da própria empresa.
Reprocessamento não deve apagar tentativas anteriores.
Reprocessamento deve gerar nova tentativa.
Reprocessamento deve registrar usuário responsável.
Reprocessamento deve respeitar regra de segurança e escopo da empresa.
```

---

## 14.5 Regras de Logs

```text
Todo evento recebido deve gerar log.
Toda ação gerada deve gerar log.
Toda execução deve gerar log.
Toda falha deve registrar erro técnico.
Toda tentativa de reprocessamento deve ser registrada.
Admin da Empresa só vê logs da própria empresa.
Payload sensível deve ser mascarado para Admin da Empresa.
Credenciais nunca devem aparecer nos logs.
```

---

## 14.6 Regras de Segurança

```text
Sistema origem precisa estar autorizado.
Empresa precisa estar ativa ou apta a operar.
Aplicação origem precisa estar ativa.
Evento precisa estar permitido para a origem.
Credenciais sensíveis não devem ser exibidas.
Payloads sensíveis devem ser mascarados conforme perfil.
Usuário de empresa não pode ver eventos de outra empresa.
Admin da Empresa não pode alterar regras globais.
Super Admin pode ver visão global.
```

Permitir que sistemas do ecossistema enviem eventos ao FP Robots.

|---|---|---:|---:|
| ROBOTS-01.01 | Receber evento de sistema origem | Alta | Sim |
| ROBOTS-01.02 | Validar origem do evento | Alta | Sim |
| ROBOTS-01.03 | Validar empresa do evento | Alta | Sim |
| ROBOTS-01.04 | Validar tipo de evento permitido | Alta | Sim |
| ROBOTS-01.05 | Bloquear evento duplicado | Alta | Sim |
| ROBOTS-01.06 | Registrar evento recebido | Alta | Sim |
| ROBOTS-01.07 | Exibir evento no painel | Alta | Sim |
| ROBOTS-01.08 | Retornar erro claro para evento inválido | Alta | Sim |

### Critérios de aceite

```text
Evento válido é recebido e registrado.
Evento inválido é recusado com motivo claro.

```

### Objetivo

# 24. Definição Comercial Resumida

```text
O FP Robots é o motor de automação e integração do ecossistema FPWebtech. Ele recebe eventos dos sistemas, aplica regras, envia e-mails, dispara webhooks, conecta sistemas internos, registra logs e permite reprocessar falhas com rastreabilidade. Seu objetivo é automatizar processos entre sistemas sem misturar as regras de negócio de cada produto.
```
