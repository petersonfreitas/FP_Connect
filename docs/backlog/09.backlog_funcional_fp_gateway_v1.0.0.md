# 1. Visão Geral

O **FP Gateway** será o módulo responsável por centralizar as integrações externas do ecossistema FPWebTech.

Ele deve concentrar:

```text
Credenciais externas
OAuth por empresa
Conexão com Mercado Pago
Pagamentos online
Futuros provedores de pagamento
WhatsApp
Meta
Webhooks externos recebidos
Normalização de respostas de provedores
Logs técnicos e operacionais de integrações
```

O objetivo central é evitar que módulos como **FP Food**, **FP Billing**, **FP Marketing**, **FP Sales** ou **FP Robots** armazenem credenciais sensíveis ou implementem diretamente regras específicas de provedores externos.

---

# 2. Objetivo do Produto

O objetivo do FP Gateway é fornecer uma camada segura e reutilizável para que qualquer módulo do ecossistema possa utilizar integrações externas sem conhecer os detalhes técnicos de cada provedor.

Exemplo:

```text
FP Food solicita criação de pagamento online.
FP Gateway executa a chamada no Mercado Pago.
FP Gateway recebe e normaliza o webhook de pagamento.
FP Gateway devolve evento padronizado para FP Food e FP Robots.
```

---

# 2.1 Diretriz de inicio aprovada

O FP Gateway deve entrar no ciclo de desenvolvimento antes de novas configuracoes avancadas de pagamento dentro do FP Food.

Diretriz operacional:

```text
Comecar com ambiente real/teste.
Usar o FP Food como primeiro consumidor.
Manter credenciais e detalhes de provedor fora do Food.
Modelar formas de pagamento do Food conforme o contrato real do Gateway exigir.
Preservar pagamento manual do Food como fallback do MVP.
Emitir eventos gateway.* para FP Robots.
```

---

# 3. Princípio Central

```text
O FP Gateway executa integrações externas.
O FP Robots orquestra eventos e automações.
Os módulos de negócio são donos das regras de negócio.
O FP Gateway não decide regra comercial, fiscal, operacional ou financeira.
```

# 6. O Que o FP Gateway Faz

O FP Gateway deve cuidar de:

```text
Cadastro de provedores externos
Cadastro de credenciais por empresa
OAuth por empresa
Validação de credenciais
Renovação ou atualização de tokens, quando aplicável
Status da conexão com cada provedor
Criação de solicitações de pagamento
Consulta de status de pagamento
Recebimento de webhooks externos
Normalização de webhooks externos
Envio de mensagens via WhatsApp, quando autorizado
Integrações Meta, quando autorizadas
Logs técnicos de chamadas externas
Logs operacionais por empresa
Reprocessamento técnico limitado
Exposição de eventos padronizados para FP Robots e módulos consumidores
```

# 8. Papéis do Sistema

## 8.1 Super Admin FPWebTech

### Pode

```text
Visualizar provedores disponíveis
Ativar ou desativar provedores globalmente
Ver empresas com integrações configuradas
Consultar logs técnicos mascarados
Apoiar reprocessamento de integrações
Ver falhas por provedor
Bloquear integração problemática
Configurar parâmetros globais permitidos
```

### Não deve

```text
Visualizar tokens em texto aberto
Alterar credenciais de empresa sem autorização
Executar regra operacional do módulo consumidor
Assumir regra comercial do FP Food, Billing ou Marketing
```

## 8.2 Admin da Empresa

### Pode

```text
Conectar conta Mercado Pago, quando disponível
Visualizar status da integração da própria empresa
Desconectar provedor da própria empresa
Autorizar integração WhatsApp, quando disponível
Autorizar integração Meta, quando disponível
Ver histórico operacional da própria integração
```

### Não deve

```text
Acessar integrações de outras empresas
Ver tokens em texto aberto
Alterar provedores globais
Acessar logs técnicos sensíveis de infraestrutura
```

## 8.3 Sistema Consumidor

Representa módulos internos como FP Food, FP Billing, FP Marketing e FP Robots.

### Pode

```text
Solicitar operação externa autorizada
Consultar status de operação criada pelo próprio módulo
Receber retorno padronizado
Receber eventos normalizados
```

### Não deve

```text
Acessar token bruto do provedor
Chamar provedor externo diretamente, salvo exceção temporária aprovada
Armazenar credenciais externas permanentes
```

# 10. Fluxo — Conectar Mercado Pago

```text
1. Admin da Empresa acessa configuração do FP Gateway.
2. Seleciona Mercado Pago.
3. Clica em conectar conta.
4. FP Gateway inicia fluxo OAuth.
5. Empresa autoriza acesso no provedor.
6. FP Gateway recebe callback.
7. FP Gateway valida credenciais.
8. FP Gateway grava tokens de forma segura.
9. FP Gateway marca provedor como conectado.
10. FP Gateway emite evento gateway.provider.connected.
11. FP Robots pode notificar responsáveis, se configurado.
```

---

# 11. Fluxo — Pagamento Online Futuro do FP Food

```text
1. Cliente final realiza pedido no FP Food.
2. FP Food calcula valor final do pedido.
3. FP Food solicita criação de pagamento ao FP Gateway.
4. FP Gateway valida se a empresa tem Mercado Pago conectado.
5. FP Gateway cria pagamento no provedor.
6. FP Gateway retorna dados normalizados ao FP Food.
7. Cliente conclui pagamento.
8. Mercado Pago envia webhook ao FP Gateway.
9. FP Gateway valida o webhook.
10. FP Gateway normaliza o status.
11. FP Gateway emite gateway.payment.approved, gateway.payment.rejected ou evento equivalente.
12. FP Food atualiza status do pedido.
13. FP Robots executa automações configuradas.
```

---

# 12. Fluxo — Envio via WhatsApp

```text
1. Um módulo interno gera evento relevante.
2. FP Robots avalia regra de automação.
3. FP Robots solicita envio externo ao FP Gateway.
4. FP Gateway valida credenciais e canal autorizado.
5. FP Gateway envia mensagem ao provedor.
6. FP Gateway registra status.
7. FP Gateway devolve resultado ao FP Robots.
8. FP Robots registra execução da automação.
```

# 13. Segurança e Credenciais

Regras obrigatórias:

```text
Tokens nunca devem aparecer em texto aberto.
Credenciais devem ser mascaradas na interface.
Logs não devem exibir payload sensível.
Cada empresa acessa apenas suas próprias integrações.
A service role nunca deve ser usada no frontend.
Toda chamada externa deve gerar log rastreável.
Toda credencial deve ter status operacional.
Desconectar provedor deve invalidar o uso futuro da credencial.
```

# 19. Regras de Multiempresa

```text
Toda integração pertence a uma empresa.
Toda operação externa deve carregar company_id.
Super Admin pode consultar globalmente.
Admin da Empresa só vê sua empresa.
Módulo consumidor só pode operar dentro do escopo da empresa.
Provedor desconectado impede novas chamadas.
Empresa suspensa pode ter integrações bloqueadas conforme regra do Admin Console/Billing.
```

---

# 20. Critérios Gerais de Aceite

```text
FP Gateway possui catálogo de provedores.
Empresa consegue conectar Mercado Pago por fluxo autorizado.
Credenciais não são exibidas em texto aberto.
FP Gateway recebe webhook e normaliza status.
FP Gateway emite eventos internos padronizados.
FP Food poderá solicitar pagamento sem conhecer detalhes do provedor.
FP Robots poderá reagir a eventos do Gateway.
Logs sensíveis são mascarados.
Cada empresa visualiza apenas suas integrações.
```

---

# 21. Riscos e Cuidados

| Risco | Mitigação |
|---|---|
| Misturar Gateway com Robots | Manter Gateway como executor externo e Robots como orquestrador |
| Acoplar Mercado Pago ao FP Food | Criar abstração de pagamento no Gateway |
| Expor tokens | Mascaramento, criptografia/segurança e auditoria |
| Webhook duplicado | Idempotência funcional por operação externa |
| Provedor fora do ar | Status claro, logs e reprocessamento |
| Crescer escopo cedo demais | Começar com Mercado Pago e estrutura reutilizável |

---

# 22. Conclusão

O **FP Gateway** deve ser tratado como módulo estratégico do ecossistema FPWebTech.

Sua criação evita acoplamento entre módulos de negócio e provedores externos, além de preparar o ecossistema para pagamentos online, WhatsApp e Meta sem duplicar código, credenciais ou regras técnicas em cada módulo.

O primeiro foco deve ser **Mercado Pago + estrutura segura de credenciais + webhooks normalizados**.
