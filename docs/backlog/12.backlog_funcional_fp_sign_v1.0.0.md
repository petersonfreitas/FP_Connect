# 1. Visão Geral

O **FP Sign** será o módulo responsável por formalizar aceite de propostas, contratos simples e documentos relacionados aos fluxos comerciais do ecossistema FPWebTech.

Seu objetivo inicial não é substituir plataformas de assinatura digital juridicamente robustas. O foco inicial é criar uma formalização operacional de aceite, com rastreabilidade, histórico e arquivamento.

---

# 2. Objetivo do Produto

O objetivo do FP Sign é permitir que propostas, contratos e termos simples gerados por módulos como **FP Sales** e **FP Billing** possam ser enviados ao cliente para aceite formal.

O FP Sign deve registrar:

```text
Documento enviado
Destinatário
Data e hora de envio
Data e hora de aceite
Status do documento
Histórico de eventos
Arquivo final aceito
Evidências básicas do aceite
```

---

# 3. Princípio Central

```text
FP Sales é dono da proposta comercial.
FP Billing é dono da cobrança SaaS.
FP Sign é dono do aceite e arquivamento formal.
FP Robots é dono das notificações e automações.
```

---

# 4. Relação com Outros Sistemas do Ecossistema

## FP Sales

Responsável por:

```text
Criar cliente/prospect
Criar oportunidade
Gerar proposta comercial
Solicitar aceite via FP Sign
Receber status de aceite
Atualizar etapa do funil conforme aceite
```

## FP Billing

Responsável por:

```text
Plano contratado
Mensalidade
Contrato comercial simples
Status financeiro da empresa
```

FP Billing pode usar FP Sign para formalizar contrato de assinatura SaaS.

## FP Robots

Responsável por:

```text
Enviar aviso de documento pendente
Reenviar lembrete
Notificar aceite concluído
Notificar expiração de link
Disparar e-mail ou WhatsApp por meio do Gateway, quando aplicável
```

## FP Gateway

Pode ser usado indiretamente quando FP Robots precisar enviar comunicação externa por WhatsApp ou outro canal previsto.

O FP Sign não deve armazenar credenciais externas.

---

# 5. O Que o FP Sign Faz

O FP Sign deve cuidar de:

```text
Receber solicitação de aceite
Criar documento de aceite
Gerar link seguro de aceite
Controlar status do documento
Registrar aceite
Registrar recusa
Registrar expiração
Gerar ou armazenar versão final do documento
Arquivar evidências básicas
Enviar eventos para FP Robots
Disponibilizar histórico para módulos de origem
```

---

# 6. O Que o FP Sign Não Faz

| Não deve fazer | Sistema responsável |
|---|---|
| Criar oportunidade comercial | FP Sales |
| Definir valor de proposta | FP Sales |
| Controlar cobrança SaaS | FP Billing |
| Processar pagamento | FP Gateway |
| Emitir nota fiscal | FP Fiscal |
| Enviar mensagem externa diretamente | FP Robots + FP Gateway |
| Executar assinatura digital avançada no MVP | Fora do escopo inicial |
| Validar força jurídica do documento | Responsável jurídico/empresa |

# 10. Fluxo — Aceite de Proposta do FP Sales

```text
1. Usuário cria proposta no FP Sales.
2. Proposta chega à etapa de envio.
3. Usuário solicita aceite via FP Sign.
4. FP Sign gera documento e link seguro.
5. FP Robots envia comunicação ao cliente.
6. Cliente acessa link.
7. Cliente visualiza documento.
8. Cliente confirma aceite.
9. FP Sign registra evidências básicas.
10. FP Sign marca documento como Aceito.
11. FP Sales recebe evento sign.document.accepted.
12. FP Sales atualiza oportunidade.
```

---

# 11. Fluxo — Recusa ou Expiração

```text
1. Documento é enviado ao cliente.
2. Cliente recusa ou prazo expira.
3. FP Sign atualiza status.
4. FP Robots notifica responsável.
5. FP Sales mantém oportunidade em negociação ou revisão.
```

---

# 12. Evidências Básicas do Aceite

O FP Sign deve registrar, quando disponível:

```text
Nome do aceitante
E-mail ou telefone informado
Data e hora do aceite
IP, quando permitido
User agent, quando permitido
Documento aceito
Versão do documento
Origem do documento
Módulo solicitante
```

Essas evidências são operacionais e não devem ser apresentadas como assinatura digital juridicamente forte no MVP.

# 17. Regras de Multiempresa

```text
Todo documento pertence a uma empresa ou cliente do ecossistema.
Usuário só acessa documentos da empresa em que atua.
Super Admin acessa conforme permissão e necessidade operacional.
Link externo deve expor apenas o documento autorizado.
Documento cancelado ou expirado não pode ser aceito.
```

---

# 18. Critérios Gerais de Aceite

```text
FP Sales consegue solicitar aceite via FP Sign.
FP Sign gera link seguro.
Destinatário consegue visualizar e aceitar documento.
Aceite gera evidência básica.
Documento aceito fica arquivado.
FP Robots recebe evento para notificação.
FP Sales recebe status atualizado.
Não há promessa de assinatura digital avançada no MVP.
```

---

# 19. Riscos e Cuidados

| Risco | Mitigação |
|---|---|
| Confundir aceite simples com assinatura digital forte | Documentar claramente o escopo do MVP |
| Documento ser alterado após aceite | Controlar versão e arquivar cópia aceita |
| Link exposto indevidamente | Usar link único, expiração e validação |
| Misturar contrato com Billing/Sales | FP Sign só formaliza e arquiva |
| Trazer gov.br cedo demais | Manter fora do MVP e avaliar futuramente |

---

# 20. Conclusão

O **FP Sign** deve ser um módulo simples e útil para formalização de aceite dentro do ecossistema FPWebTech.

Seu primeiro valor está na integração com **FP Sales**, permitindo transformar proposta aceita em evidência operacional e contrato simples arquivado, sem exigir assinatura digital avançada no MVP.
